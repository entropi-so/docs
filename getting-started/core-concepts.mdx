## 1. The optimistic pipeline

`POST /pipeline/analyze` does not wait for the work to finish. It writes a `peeking` row to the database, schedules a background task, and returns a `201 Created` in under 500&nbsp;ms.

The pipeline then runs OCR, crop, and search enrichment asynchronously and updates the row as it progresses. Clients poll `GET /products/{id}` to observe the status changes.

This is a deliberate trade-off. The user-perceived latency drops from ~10&nbsp;s to under half a second. The cost is that clients must poll, and a server restart can orphan in-progress products. See [the durability caveat](/api-reference/analyze#durability-caveat-mvp).

## 2. The status lifecycle

Every product moves through a fixed state machine.

| Status      | Meaning                                                                   |
| ----------- | ------------------------------------------------------------------------- |
| `peeking`   | Row created. OCR + bbox call (Gemini 2.5 Flash) is running.               |
| `framing`   | OCR succeeded. Local Pillow crop is running. Near-instant (&lt;100&nbsp;ms). |
| `sleuthing` | Crop done. Merchant search and scraping are running. 15&nbsp;s hard cap.  |
| `ready`     | Pipeline complete. Product fields populated.                              |
| `stumped`   | Pipeline failed. `error_message` describes why.                           |

`ready` and `stumped` are terminal. Stop polling once you see them.

## 3. Graceful degradation

Each stage of the pipeline has its own failure policy:

- **OCR failure** is fatal. The product ends in `stumped`.
- **Crop failure** falls back to the original (uncropped) image. The pipeline continues.
- **Search enrichment failure** falls back to OCR-only data. The product still reaches `ready` — just without a `product_url` or scraped metadata.

Search enrichment is wrapped in a 15&nbsp;s `asyncio.wait_for` and protected by a circuit breaker (3 consecutive failures opens it for 60&nbsp;s, during which enrichment is skipped entirely). Low-confidence enrichment results (&lt;90%) are discarded but do **not** trip the breaker.

The takeaway: a `ready` status does not guarantee enriched data. Inspect the `tags` field — the absence of `search_confidence:<n>` means the product fell back to OCR-only.

## 4. Services as stages

The pipeline is four stateless services chained together:

<CardGroup cols={2}>
  <Card title="Image Service" icon="image" href="/services/image-service">
    Decode, convert (HEIC&nbsp;→&nbsp;JPEG), resize, base64-encode.
  </Card>
  <Card title="OCR + Bbox" icon="eye" href="/services/ocr-service">
    Single Gemini 2.5 Flash call returning both product fields and a bounding box.
  </Card>
  <Card title="Crop Service" icon="crop" href="/services/crop-service">
    Local Pillow crop with 20% padding. No AI call.
  </Card>
  <Card title="Search Enrichment" icon="magnifying-glass" href="/services/search-enrichment">
    Gemini grounded search + scraping, behind a timeout and a circuit breaker.
  </Card>
</CardGroup>

Each service is async, stateless, and independently testable. See [Architecture](/architecture) for the full data flow and [Run tests](/guides/run-tests) for how the project structures pytest vs. dev scripts.

## Mental model summary

You should now be able to answer four questions without re-reading anything:

1. Why does `POST /pipeline/analyze` return so fast?
2. What does `status: stumped` tell you that `status: ready` does not?
3. When does enrichment get skipped, and how does the client detect it?
4. Which stages of the pipeline call an AI provider, and which run locally?

If any of those is unclear, re-read the relevant section above before moving on.
