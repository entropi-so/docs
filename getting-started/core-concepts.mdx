## 1. The synchronous pipeline

`POST /pipeline/analyze` runs the whole capture **in the request** and returns the finished product in the `201 Created` response body. There is no background task and no polling: image prep, visual understanding, crop, and embedding all complete before the response is sent.

The row is created already in its terminal state — `ready` on success, or `stumped` (best-effort, keeping the raw photo) on failure. Perceived speed is handled on the client (a splash screen during the request), so the backend can stay simple, linear, and in-process.

Concurrency is real, not serialized: the route is async, every external call is awaited, and the CPU-bound image work (Pillow decode/crop/resize) runs in worker threads via `asyncio.to_thread`. N simultaneous captures progress in parallel on the event loop. A per-request timeout (45&nbsp;s) caps the understanding step.

## 2. The status lifecycle

A captured product is created directly in a terminal state.

| Status    | Meaning                                                                            |
| --------- | ---------------------------------------------------------------------------------- |
| `ready`   | Capture complete. The record is fully populated.                                   |
| `stumped` | Understanding failed or timed out. The raw photo is kept in `image_url`; `error_message` explains why. |

<Note>
  The statuses `peeking`, `framing`, and `sleuthing` belonged to the old
  optimistic/polling pipeline. They are **deprecated and never emitted**.
  Clients only ever see `ready` or `stumped` (plus `purchased` once an item
  is marked bought).
</Note>

## 3. The capture record

The whole point of a capture is a rich, search-native record — useful even when the exact product is unknown. An unidentified jacket photographed in a store should still be saved with enough meaning (object, colors, style) to live in a moodboard and resurface on a query like *"cozy green jacket"*.

Every record carries:

- **`content_type`** — the coarse bucket: `place`, `clothing`, `decor`, `food`, or `other`.
- **`description`** — a rich, natural description of what is visible. This is the main driver of search quality.
- **`category`**, **`object_type`** — e.g. `outerwear` / `wool jacket`.
- **`tags`** — colors, style descriptors, and setting, folded into one comma-separated field.
- **`merchant`** + **`brand_confidence`** — the brand **only when it is actually readable** in the image (a logo, a label, visible text); otherwise both are `null`. Brand is never guessed from style.
- **`raw_text`** — verbatim text read in the image.
- **`image_url`** — the cropped product image (base64 data-URI).

There are no web fields: `product_url` is always `null` — capture involves zero web calls.

## 4. Semantic search comes for free

Every `ready` capture is embedded (best-effort) into a multimodal vector at capture time. [`GET /products/search`](/api-reference/search) then embeds a free-text query and ranks the caller's products by cosine similarity — matching on what the photo *shows* (color, object, setting) as much as on its text. So *"veste verte cosy"* finds a green jacket even if none of those words are stored.

If the embedding provider (Voyage) is not configured, capture still works — the product is simply left unindexed and search returns `503`.

## Services as stages

The pipeline is four stateless services chained together:

<CardGroup cols={2}>
  <Card title="Image Service" icon="image" href="/services/image-service">
    Decode, convert (HEIC&nbsp;→&nbsp;JPEG), resize, base64-encode.
  </Card>
  <Card title="Vision Service" icon="eye" href="/services/vision-service">
    A single Gemini 2.5 Flash multimodal call: recognition + text reading + structured record + bounding box.
  </Card>
  <Card title="Crop Service" icon="crop" href="/services/crop-service">
    Local Pillow crop with 20% padding. No AI call.
  </Card>
  <Card title="Embedding Service" icon="vector-square" href="/services/embedding-service">
    Voyage multimodal embedding (text + cropped image) for semantic search.
  </Card>
</CardGroup>

Each service is async, stateless, and independently testable. See [Architecture](/architecture) for the full data flow and [Run tests](/guides/run-tests) for how the project structures pytest vs. dev scripts.

## Mental model summary

You should now be able to answer these without re-reading anything:

1. Why does `POST /pipeline/analyze` return the finished product instead of a `product_id` to poll?
2. What does `status: stumped` guarantee about the response (hint: the raw photo)?
3. When is `merchant` populated, and when is it `null`?
4. Which stages call an AI provider, and which run locally?

If any is unclear, re-read the relevant section above before moving on.
