## Overview

The **Embedding Service** indexes a product into a single 1024-dim multimodal
vector that reflects **both** what the photo shows (color, object, setting)
and the record's text. That is what lets a query like *"chocolat"* match a
chocolate-colored sofa whose text never uses the word.

It runs at the end of capture (best-effort) and is also used by the
retroactive backfill, so live and backfilled products share the exact same
indexing path.

**Directory**: `app/services/embedding/`

## Functions

<ParamField body="build_product_text" type="function">
  Composes the textual side of the embedding from the record's fields, in
  priority order (`name`, `content_type`, `object_type`, `category`, `tags`,
  `merchant`, `description`), skipping empties.
</ParamField>

<ParamField body="compute_product_embedding" type="async function">
  Embeds the composed text + cropped image into one document-side vector.
</ParamField>

```python
def build_product_text(fields: Mapping[str, Any]) -> str

async def compute_product_embedding(
    text: str | None,
    image_base64: str | None,

```

<ParamField body="compute_query_embedding" type="async function">
  Embeds a free-text search query into a query-side vector, with an LRU cache.
  Same model as the document side — query and product vectors live in the same
  space.
</ParamField>

```python
async def compute_query_embedding(text: str) -> list[float]
```

## Storage

Embeddings are stored in the `products.embedding` column (`pgvector`,
`Vector(1024)`, nullable). Search ranks with the pgvector cosine-distance
operator (`<=>`), pushing the ordering, `limit`, and `offset` into Postgres —
only the top-k rows come back, never a full-catalog sort in memory.

## Best-effort at capture

In the pipeline, indexing runs **after** the product is otherwise complete and
any failure is swallowed:

```python
try:
    text = build_product_text(record)
    embedding = await compute_product_embedding(text, cropped_image)
    # persisted alongside the product
except Exception:
    # left unindexed; product stays `ready`, picked up by the backfill later
    ...
```

So a missing `VOYAGE_API_KEY` or a transient error never breaks a capture — it
just leaves the product out of search until it is re-indexed.

## Query caching

`compute_query_embedding` keeps an LRU cache keyed by the normalized query.
Search-as-you-type sends one request per keystroke, so re-embedding the same
or repeated text is the latency/cost/rate-limit hotspot — the cache
neutralizes it. A cache hit returns the exact vector a miss would have
produced.

## Backfill

```bash
python scripts/backfill_embeddings.py
```

Embeds every `ready` product that has no vector yet. Use it after enabling
`VOYAGE_API_KEY` for the first time, or to index products created before
search existed.

## Configuration

```bash
VOYAGE_API_KEY=your_voyage_api_key_here
```

Optional. Without it, capture still works; [`GET /products/search`](/api-reference/search)
returns `503` and products are left unindexed.
