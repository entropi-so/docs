## Context

`POST /pipeline/analyze` returns a `product_id` immediately. The actual pipeline runs in the background. You retrieve the result by polling `GET /products/{id}` until you see a terminal status.

This guide covers the polling pattern. For the response shape, see [`GET /products/{id}`](/api-reference/products).

## Stop conditions

Stop polling as soon as `status` is one of:

- `ready` — pipeline complete, product fields populated.
- `stumped` — pipeline failed, `error_message` populated.

Both are terminal. Further polling returns the same payload.

## Recommended cadence

<CardGroup cols={2}>
  <Card title="In-app camera capture" icon="camera">
    Poll every **300&nbsp;ms**. The user is staring at a loading state; perceived latency matters.
  </Card>
  <Card title="Share-sheet or gallery import" icon="images">
    Poll every **500&nbsp;ms to 1&nbsp;s**. The user is on a list view, not waiting on the pipeline.
  </Card>
</CardGroup>

End-to-end pipeline duration is typically 6&ndash;14&nbsp;s. Anything beyond 20&nbsp;s without a terminal status is a bug — file an issue with the `product_id`.

## Reference loop

```bash
PID=42
while :; do
  resp=$(curl -s "http://localhost:8000/products/$PID" \
    -H "Authorization: Bearer $TOKEN")
  status=$(echo "$resp" | jq -r .status)
  echo "$(date +%H:%M:%S) $status"
  [[ "$status" == "ready" || "$status" == "stumped" ]] && echo "$resp" | jq && break
  sleep 0.5
done
```

## Steps

<Steps>
  <Step title="Capture the product_id">
    Returned by `POST /pipeline/analyze` in the `id` field. Persist it client-side immediately.
  </Step>
  <Step title="Schedule a poll loop">
    Use a backoff that matches the user context (300&nbsp;ms or 500&nbsp;ms&ndash;1&nbsp;s).
  </Step>
  <Step title="Stop on a terminal status">
    Break the loop on `ready` or `stumped`. Do not keep polling.
  </Step>
  <Step title="Render the result">
    For `ready`, render the product fields. For `stumped`, render `error_message` and the [failure recovery flow](/guides/handle-failures).
  </Step>
</Steps>

## Expected result

A successful poll produces a sequence like:

```
13:42:01 peeking
13:42:02 peeking
13:42:04 framing
13:42:04 sleuthing
13:42:11 ready
```

## Common mistakes

- **Continuing to poll after `ready` or `stumped`** — wastes requests; both are terminal.
- **Polling faster than 300&nbsp;ms** — pipeline stages do not transition that fast; you are only generating server load.
- **Ignoring `error_message` on `stumped`** — it always contains useful diagnostic information.
- **Hard-coding a single timeout** — most pipelines finish in 6&ndash;14&nbsp;s, but enrichment can take up to 15&nbsp;s on its own. Allow 30&nbsp;s before treating absence of a terminal status as a hang.
