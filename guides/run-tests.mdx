## Context

Entropi splits tests into two locations with different audiences:

- `tests/` — pytest suite, runs in CI, mocks every AI client.
- `scripts/dev_tests/` — standalone scripts you run by hand against real provider APIs.

The two trees never overlap. Pytest must never call a real Gemini or Groq endpoint. Dev scripts always do.

## Steps

<Steps>
  <Step title="Activate the virtual environment">
    ```bash
    source venv/bin/activate
    ```
  </Step>
  <Step title="Install dev dependencies">
    ```bash
    pip install -r requirements-dev.txt
    ```
  </Step>
  <Step title="Run the pytest suite">
    ```bash
    pytest tests/
    ```

    Useful flags:

    ```bash
    pytest tests/test_pipeline_api.py -v      # one file
    pytest tests/ -k search                    # filter by name
    pytest tests/ -x                          # stop at first failure
    ```

    Run without `VOYAGE_API_KEY` set (or mock the embedding call); a live
    Voyage key can hang the suite.
  </Step>
  <Step title="Run a dev script when iterating on AI">
    ```bash
    python scripts/dev_tests/unit/vision_service/test_vision_extraction.py
    ```

    Dev scripts hit real providers and consume budget. Use small inputs.
  </Step>
</Steps>

## What to write where

<CardGroup cols={2}>
  <Card title="Add a pytest test when..." icon="check">
    You add or change an HTTP endpoint, an auth rule, or a DB-level behavior.
  </Card>
  <Card title="Add a dev script when..." icon="flask">
    You add or tune an AI service (prompt change, model swap, scraping rule). The goal is to inspect real provider output, not to assert pass/fail.
  </Card>
</CardGroup>

The project convention: every new endpoint ships with a pytest test; every new service ships with a dev script. Both can apply to the same change.

## Test infrastructure

`tests/conftest.py` provides two fixtures used everywhere:

- `db_session` — a fresh SQLAlchemy session backed by an in-memory SQLite database. Tables are created and dropped per test.
- `client` — a `TestClient` with FastAPI's `get_db` dependency overridden to use `db_session`, so HTTP-level tests share state with direct DB queries.

`pytest.ini` enables `asyncio_mode = auto`, so `async def` tests are collected automatically.

## CI execution

The GitHub Actions workflow at `.github/workflows/test.yml` runs the suite on every push to `main` and every pull request. Required environment variables are dummy values — Pydantic validates them at startup but the AI clients are mocked at runtime.

<Warning>
  A test that hits a real Gemini or Groq endpoint will fail in CI (rate limits, no internet) and burn provider budget. Keep those interactions in `scripts/dev_tests/`.
</Warning>

## Common mistakes

- **Writing a pytest test that imports a real LLM client** — mock it with `monkeypatch` against the shared client in `app/core/llm_clients.py` or the service module that imports it.
- **Adding a dev script under `tests/`** — pytest will collect it and call real APIs in CI.
- **Skipping the dev script for a new AI service** — there is no other way to inspect prompt and scraping behavior end-to-end.
- **Checking dev script output into version control** — they print to stdout for live inspection; do not snapshot the output.
