## Context

Every capture is indexed into a multimodal embedding at save time. [`GET /products/search`](/api-reference/search) embeds a free-text query and ranks your products by cosine similarity, so a search matches on what the photo *shows* as much as on its text. *"veste verte cosy"* surfaces a green jacket even when none of those words are stored.

This is text-in, product-out. It is never a reverse image search.

## Steps

<Steps>
  <Step title="Authenticate">
    Search is scoped to your own products, derived from the JWT.

    ```bash
    TOKEN=$(curl -s -X POST http://localhost:8000/auth/login \
      -H "Content-Type: application/json" \
      -d '{"username":"you@example.com","password":"..."}' \
      | jq -r .access_token)
    ```
  </Step>
  <Step title="Query">
    ```bash
    curl -s -G "http://localhost:8000/products/search" \
      -H "Authorization: Bearer $TOKEN" \
      --data-urlencode "q=cozy green jacket" \
      --data-urlencode "limit=20" | jq
    ```
  </Step>
  <Step title="Render, sorted by relevance">
    Results come back most-relevant first, each with a `relevance` score. There
    is no server-side cutoff, so a live gallery can reorder as the user types
    without ever blanking out.
  </Step>
</Steps>

## Search-as-you-type

The endpoint is built for a debounced request per keystroke:

- **Query embeddings are cached** (LRU, keyed by the normalized query), so repeated/growing queries do not re-embed the same text.
- **No threshold** — always the top `limit` by relevance, so results never disappear mid-typing.
- Debounce on the client (~150–250&nbsp;ms) to avoid a request per character.

## What makes a product findable

Search quality comes mostly from the **`description`** and the cropped image
that were embedded together. Colors, style, and setting are also folded into
`tags`, which contributes to the text side of the embedding. A rich capture
record (even for an unidentified item) is what makes *"green cozy jacket"*
work.

A product is only searchable once it has an embedding. Captures are embedded
best-effort at save time; if that step was skipped (e.g. no `VOYAGE_API_KEY`
at the time, or a transient error), the product is `ready` but excluded from
search until re-indexed.

## Backfill for unindexed products

Products created while embedding was unavailable, or before search existed,
can be indexed retroactively:

```bash
python scripts/backfill_embeddings.py
```

It embeds every `ready` product that has no vector yet, using the same
`build_product_text` + `compute_product_embedding` path as the live pipeline,
so live and backfilled products rank consistently.

## Error responses

- **422** — `q` empty/whitespace or `limit`/`offset` out of bounds.
- **503** — `VOYAGE_API_KEY` not configured. Capture still works; search is unavailable until a key is present.
