## Context

Entropi's capture has two failure surfaces:

1. **Request errors** — bad auth, wrong list, oversized file. Returned as a 4xx/5xx HTTP response. No product is created.
2. **Capture failures** — understanding could not complete. **Not** an HTTP error: the product is still created and returned in the `201` body as `status: "stumped"`, with the raw photo kept in `image_url` and a human-readable `error_message`.

Treat them as different classes: the first is a caller mistake, the second is an outcome you display.

## Steps

<Steps>
  <Step title="Inspect the HTTP status">
    Anything that is not `201 Created` from `POST /pipeline/analyze` is a
    request error. See [Failure modes](/api-reference/analyze#failure-modes).
  </Step>
  <Step title="On 201, check the product status">
    The body is the finished product. If `status` is `ready`, render it. If it
    is `stumped`, handle it as below.
  </Step>
  <Step title="On stumped, read error_message and keep the photo">
    The message is engineer-readable. The raw photo is in `image_url`, so the
    capture is never lost — show it with a soft "we could not read this one"
    state.
  </Step>
  <Step title="Decide whether to retry">
    Some failures are user-actionable (retake a clearer photo). Others are
    infrastructure (provider outage). The message tells you which.
  </Step>
</Steps>

## Request errors

| Status | Cause                                                        | Client action                                |
| ------ | ----------------------------------------------------------- | -------------------------------------------- |
| 401    | Missing or invalid bearer token.                            | Re-authenticate.                             |
| 403    | The `list_id` does not belong to the authenticated user.    | Use a list owned by the caller.              |
| 404    | The `list_id` does not exist.                               | Create or pick a valid list.                 |
| 413    | File exceeds 10&nbsp;MB.                                    | Compress or resize before resubmitting.      |
| 500    | Failure to persist the product row (rare — DB issue).       | Retry with backoff. Escalate if it persists. |

## Capture failures (`stumped`)

A product ends in `stumped` when understanding returns nothing, the image
cannot be decoded, or the 45&nbsp;s cap is hit. The row is still created and
returned:

```json
{
  "id": 42,
  "status": "stumped",
  "error_message": "Capture failed: Visual understanding returned no data.",
  "image_url": "data:image/jpeg;base64,/9j/...",
  "name": null,
  "board_ids": [1],
  "created_at": "2026-07-04T02:39:26.359572-07:00"
}
```

Common `error_message` values:

- `Capture failed: Visual understanding returned no data.` — the model could not make sense of the image.
- `Image conversion failed: <details>` — unsupported codec or corrupted bytes.
- `Capture timed out` — the pipeline hit the 45&nbsp;s cap.

Because the raw photo is preserved, a good UX keeps the `stumped` product in
the list and lets the user retry or annotate it, rather than discarding it.

## Graceful degradation (no failure)

Some sub-steps degrade silently and still yield a `ready` product:

- **Crop failure** falls back to the original (uncropped) image.
- **Embedding failure** leaves the product `ready` but unindexed — it will not appear in [semantic search](/guides/semantic-search) until the backfill re-indexes it. Everything else is unaffected.

## Common mistakes

- **Retrying a `413` automatically** — the file is too large. Resize it client-side first.
- **Discarding a `stumped` product** — the raw photo is still there; keep it and offer a retry.
- **Surfacing raw `error_message` to end users** — it is engineer-readable. Map it to a UX-appropriate message.
- **Assuming `ready` means indexed** — if the embedding step was skipped, the product is `ready` but not yet searchable.
