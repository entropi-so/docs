## 2026-07 â€” Synchronous capture pipeline + semantic search

`POST /pipeline/analyze` is now **synchronous**: it runs the whole capture in the request (image prep â†’ visual understanding â†’ crop â†’ embedding) and returns the finished product. The optimistic/polling machinery and the web-enrichment step are gone; there are **zero web calls** in the capture path.

Products are created directly in `ready`, or `stumped` (best-effort, keeping the raw photo). The single Gemini 2.5 Flash call now produces a **lifestyle record** â€” `content_type`, a rich `description`, `category`, `object_type`, colors/style/setting (folded into `tags`), `brand` + `brand_confidence` (only when readable), and `raw_text`. The `ocr_service` was renamed to `vision_service` (`analyze_capture`).

Every capture is indexed into a multimodal embedding, powering the new [`GET /products/search`](/api-reference/search) semantic search.

**Migration notes**:

- **Stop polling.** `POST /pipeline/analyze` now returns the finished product. Remove any `GET /products/{id}` poll loop.
- **Statuses** `peeking`, `framing`, and `sleuthing` are deprecated and never emitted. Only `ready` / `stumped` / `purchased` occur. Drop them from client enums.
- **New fields** on products: `content_type`, `brand_confidence`, `raw_text`. `product_url` is always `null` (no web).
- **Batch** (`POST /pipeline/analyze/batch`) now returns a list of finished products (not `{id, status}` stubs) plus `errors[]`.
- **Config**: `GEMINI_SEARCH_API_KEY` is removed; add the optional `VOYAGE_API_KEY` to enable search. Existing products can be indexed with `python scripts/backfill_embeddings.py`.
- **DB**: three new nullable columns (`content_type`, `brand_confidence`, `raw_text`) via an additive migration.

## 2026-04-25 â€” I037: Optimistic pipeline

`POST /pipeline/analyze` is now optimistic. It commits a `peeking` row and returns a `201` in under 500&nbsp;ms; the full pipeline (OCR, crop, enrichment) runs in the background and is observable via `GET /products/{id}` polling.

Concurrent submits also run in real parallel â€” every blocking SDK call (Gemini, Pillow) is wrapped in `asyncio.to_thread`.

`GET /products/{id}` is now the canonical polling endpoint. See [`api-reference/products`](/api-reference/products).

**Migration notes**:

- Clients that relied on `POST /pipeline/analyze` returning a fully populated product must switch to a poll loop. *(Superseded by the 2026-07 synchronous pipeline â€” polling was later removed entirely.)*
- The status field can now be `peeking`, `framing`, `sleuthing`, `ready`, or `stumped`. Update any client-side enums.

## 2026-04 â€” I032: Unified OCR + bbox

OCR and bounding-box extraction collapsed into a single Gemini 2.5 Flash call (`extract_and_locate_product`), saving ~3&nbsp;s per pipeline run. Crop service is now a pure local Pillow operation with no AI call.

**Migration notes**: none for API consumers. Internal services calling the old two-step OCR/bbox interface must update to the unified call.
