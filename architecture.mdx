## Overview

Entropi follows a **synchronous pipeline architecture**: each service processes the data sequentially, transforming a raw capture into a structured, search-indexed product record — all within a single HTTP request. There is no background task, no queue, and no polling.

## Detailed Pipeline

<Steps>
  <Step title="1. Image Service">
    **Input**: Raw image bytes (HEIC/JPEG/PNG/WebP)
    **Output**: Base64-encoded JPEG
    **Purpose**: Convert and optimize images for AI processing

    - Converts HEIC → JPEG
    - Resizes to max 2048×2048px
    - Optimizes quality (95% JPEG)
    - Encodes to base64

    Runs before the timeout window so the decoded raw photo is always
    available for the best-effort `stumped` fallback.
  </Step>

  <Step title="2. Vision Service (single Gemini call)">
    **Input**: Base64 image
    **Output**: Structured lifestyle record (JSON) + normalized bbox
    **Purpose**: Understand the image in one shot

    - One Gemini 2.5 Flash multimodal call via `analyze_capture()`
    - Object recognition + text reading + structured JSON + bounding box
    - Returns `content_type`, `description`, `category`, `object_type`,
      `colors`, `style_tags`, `setting`, `brand` + `brand_confidence`
      (only when readable), `raw_text`, `price`, and
      `bbox: [ymin, xmin, ymax, xmax]` (0–1000 normalized, nullable)
    - No OCR engine or object detector on the side, no web
  </Step>

  <Step title="3. Local Crop">
    **Input**: Base64 image + bbox from step 2
    **Output**: Cropped base64 image
    **Purpose**: Focus on the main subject

    - Pure local Pillow operation (`crop_with_bbox()`) — no AI call
    - Crops with 20% padding (configurable)
    - Falls back to the original image on any failure (graceful degradation)
  </Step>

  <Step title="4. Embedding (best-effort)">
    **Input**: Composed record text + cropped image
    **Output**: A 1024-dim multimodal vector
    **Purpose**: Index the product for semantic search

    - One Voyage multimodal call (`compute_product_embedding()`)
    - Combines the record's text and the image into a single vector
    - Best-effort: a failure here leaves the product `ready` with a null
      embedding, to be picked up later by the retroactive backfill
  </Step>

  <Step title="5. Database Storage">
    **Input**: The finished record (+ embedding)
    **Output**: Product row in its terminal state
    **Purpose**: Persist the capture

    - Inserts one `products` row already in `ready` (or `stumped`)
    - Attaches it to the target list
    - Returns the full product object in the HTTP response
  </Step>
</Steps>

## Error Handling

Each service degrades gracefully; only a failure of understanding produces `stumped`:

<AccordionGroup>
  <Accordion title="Image Service failures">
    If the image cannot be decoded, the product is persisted as `stumped`
    with an `error_message`. The response is still `201` — capture never
    returns a blocking 5xx for a bad image.
  </Accordion>

  <Accordion title="Vision Service failures / timeout">
    If understanding returns nothing or the 45&nbsp;s cap is hit, the product
    is persisted as `stumped`, **keeping the raw photo** in `image_url`.
  </Accordion>

  <Accordion title="Crop failures">
    Falls back to the original image. The pipeline continues; status is
    still `ready`.
  </Accordion>

  <Accordion title="Embedding failures">
    Swallowed. The product stays `ready` with a null embedding and is
    re-indexed later by the backfill. Search simply excludes unindexed rows.
  </Accordion>
</AccordionGroup>

## Performance Considerations

Each pipeline run is sequential (every step consumes the previous step's output), but **concurrent captures run in real parallel**. All blocking SDK calls (Gemini, Voyage) are awaited and the Pillow work is offloaded via `asyncio.to_thread`, so N simultaneous `/pipeline/analyze` requests do not serialize on the event loop. Batch imports fan out concurrently (bounded).

**Typical processing times** (per capture):
- Image conversion: ~200&nbsp;ms
- Vision understanding (single Gemini call): ~3–5&nbsp;s
- Local Pillow crop: &lt;100&nbsp;ms
- Embedding: ~1–2&nbsp;s (best-effort)
- **Total**: ~4–8&nbsp;s end-to-end, hard-capped at 45&nbsp;s

<Tip>
  Perceived latency is handled on the client (a splash screen during the
  request), so the backend stays optimistic, linear, and in-process — no
  queue, Celery, or Redis.
</Tip>
